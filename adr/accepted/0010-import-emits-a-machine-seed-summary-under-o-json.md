# ADR-0010: Import emits a machine seed summary under -o json

> State: Accepted

## Status

Accepted

## Stakeholders

- Maintainer — Brett Fowle

## Context and Problem Statement

`adroit import --from-assessment` is the ingest seam: it turns an assessment
export into a proposed-ADR backlog. Downstream loop runners (the assessments
app's seam-check, the Adopt-stage engine) need to *assert* an ingest result —
what was seeded, what the dedupe guard skipped — but `import` only produced a
human report. Worse, the global `-o json` flag was **silently ignored**: the
verb accepted it, emitted human text on stdout, and exited 0, so a consumer
piping stdout into a JSON parser failed at the parse, not at the CLI — verified
live before this change. Every other verb that advertises a `json_output` shape
in the manifest honors the flag, so the ignore was also a manifest-contract
gap.

The shape of the summary has to be settled once: it becomes a cross-repo
contract the moment the first consumer parses it.

## Decision Drivers

- Stdout purity: under `-o json`, stdout must be exactly one parseable JSON
  document on every success path (including "nothing to seed"); human notes go
  to stderr — the existing convention for every JSON-emitting verb.
- Fit the house view-type conventions: a serde struct in `src/view.rs`,
  schemars-derived, registered in the manifest's `types`, named in the
  `classified()` table's `json_output` column.
- Truthfulness: report only facts the run actually established — a `--dry-run`
  allocates no identity, and identity isn't predictable under every naming
  scheme.
- Additive only: `manifest_schema` stays 1; nothing existing changes shape.
- No duplicated tallies that can drift from the data they summarize.

## Considered Options

1. **A dedicated `view::ImportSummary` envelope** — `{source, assessment,
   dry_run, seeded: [{reference, title, status, domain}], skipped: [titles]}`;
   `reference` is `null` under `--dry-run`; counts are the array lengths.
2. **Reuse `AdrSummary[]`** — emit the seeded ADRs as the existing list shape.
3. **A flat counters object** — `{created: N, skipped: N}` only.

## Decision Outcome

Chosen: **Option 1 — a dedicated `ImportSummary` envelope**, because the run
itself (not just its products) is the artifact a loop runner asserts: dry-run
vs wet, provenance (`source` file, `assessment` name), and the skip list are
run facts no per-ADR shape can carry. Option 2 cannot represent a dry run (no
ADRs exist to summarize), drops the skipped titles, and would force the
loaded-row contract onto entries that were never read back from disk. Option 3
hides the titles, which are exactly what the assessments seam-check pins.

Within the envelope: `reference` is `Option` and `null` under `--dry-run` —
identity is allocated only on write, so a preview truthfully carries none
rather than a guess that breaks under the date/uuid schemes. `status` is the
typed `Status` (serialized `"Proposed"`, matching `status -o json`) and carries
the configured `default_status` rather than a hardcoded constant. `skipped`
stays a list of titles — the only skip reason today is the dedupe guard, and a
richer object would be speculative schema. Counts are derived (`seeded.len()`,
`skipped.len()`), never stored alongside.

`import` thus becomes the one **write** verb with a `json_output` entry. That
is additive and safe: the MCP read-only projection filters on
`reads && !writes` ([ADR-0005](./0005-mcp-server-projects-only-read-only-manifest-verbs.md)),
so `import` is still never projected; the `--output` long help and its guard
test now enumerate it alongside the read verbs.

The envelope is pinned cross-repo by a vendored golden fixture
(`tests/fixtures/golden-assessment.yaml`, generated by the assessments app's
real exporter in an assessments checkout — sibling or clone, per the suite
resolution convention — and re-vendored verbatim) so contract drift on either
side fails CI here.

### Positive Consequences

- A loop runner asserts ingest results mechanically (counts, titles, skips) —
  no prose scraping; the silent-ignore bug class is closed for `import`.
- The shape is discoverable and validatable: `manifest -o json` carries the
  `ImportSummary` schema like every other `json_output` type.
- Dry-run and wet runs share one shape, so a consumer's parser covers both.

### Negative Consequences

- The envelope is now a contract: renaming or restructuring its fields is a
  breaking change for downstream consumers (additive evolution only).
- `reference: null` on dry-run means a preview cannot tell a consumer which
  identifiers a wet run will assign — deliberate, but it forces two-step flows
  that want the references to actually import.
- A second `json_output` convention ("write verbs may report") now exists with
  a single member; if more write verbs grow reports, the pattern should be
  consolidated rather than re-invented per verb.

## Implementation

Landed with this decision (milestone M4): `view::ImportSummary` +
`view::ImportSeed`; `cmd_import` takes the global `OutputFormat`, routes every
human stdout line behind it, and emits the envelope on all success paths;
`classified()` gains `json_output: Some("ImportSummary")` and the manifest
`types` registers the schema; the `--output` long help names `import`; CLI
tests cover wet/dry/skip/empty/`--ai` JSON runs plus the vendored golden
contract test; docs synced (`automation.md`, `workflow.md`, `reference/cli.md`,
`dev/testing.md`).
